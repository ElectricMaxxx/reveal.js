HTTP

HTTP, so kennen wir es als Entwickler ist das sog. HyperText Transfer Protocol.

HTTP hat etwas von der Konversation mit dem Guten alten Brief:
Ich sende eine Information an eine bestimmte Addresse und erhalte eine Antwort. Sowahr der Empfänger das will :-). Jeder Brief enthält Meta-Informationen für den Versand auf dem Umschlag. Diese entsprechen den Headern bei einem HTTP-Request oder -Response. Dazu enthält jeder Brief Inhalt - der sog. Body.
D.h. es geht um einen Request an einen Server und desen Response.
Dann schauen wir uns einmal einen solchen Request an:
KLICK

Host: www.expamle.dev
Content-Type: text/html

Status Codes

  • 1XX - Bearbeitung dauert an (Pending)
  • 2XX - Anfrage Erfolgreich
  • 3XX - Umleitungen
  • 4XX - Fehler in der Anfrage/beim Client
  • 5XX - Fehler am Server
Die Status Codes ...
... geben Information über Erfolg oder Misserfolg eines Requests.
... werden durch eine Phrase ergänzt, die den Grund (bspw. für den Misserfolg) darstellt.
... und werden in Gruppen unterteilt.

KLICK Alle die mit einer 1 beginnen signalisieren, dass der Server noch etwas Zeit benötigt.
KLICK Die 2 am Anfang stellt einen Erfolg dar. Bekanntester Vertreter ist: 200 OK
KLICK Die 3 ist nicht kritisch, aber es läuft schon nicht alles nach Plan. Beschreibt Umleitung. Bsp: 301 Moved Permanently
KLICK Die 4 markiert einen Fehler beim der Client Applikation. Bsp: 404 Not Found - eine Resource wurde angefragt, die es nicht gibt.
KLICK Die 5 am Anfang signalisiert ein Fehler am Server. Bsp: 500 Internal Server Error

POST

POST dient dem Anlegen einer neuer Resource. Im Grunde ergänzt es eine Liste an Resourcen um einen neuen Eintrag.
Vor allem verwendet man POST wenn man sich nicht ganz sicher ist unter welcher URL der neue Eintrag anglegt werden soll. Denn meist ist die ID nicht bekannt, da der Server diese bspw. über ein auto-increment zuteilt.
Natürlich muss ich im Body die Daten zu der neuen Resource mit senden. 
                        
POST /users HTTP/1.1
Host: www.example.dev
Content-Type: application/json

{
    "firstName": "Max",
    "lastName": "Berghoff"
}

POST request

- ändert aber nix
- Angabe Content-Type gibt an von welchem Type dei gesendeten Daten sind 

                        
HTTP/1.1 201 Created
Content-Type: application/json;
Content-Location: /users/3

{
    "id": "3",
    "firstName": "Max",
    "lastName": "Berghoff"
}

POST response

- hat wohl geklappt
- Content-Location: Da findet man jetzt den neu angelegten Usr
- Body = User zurück mit seinder

PATCH

- Zum verändern von einzelnen Werten einer Resource - macht weniger kaput wie PUT

POST / PUT / PATCH

- definitiv nicht idempotent. Ist die Resource noch da?
- haben Side-Effects
- ändern schließlich Werte oder legen neue Resourcen an

DELETE

- zum entfernen einer Resource
- natürlich mit Side-Effects 
                        
DELETE /users/3 HTTP/1.1
Host: www.example.dev

DELETE request

- Resource muss natürlich bekannt sein
- kein Content-Type nötig

DELETE response

- hat wohl geklappt
- kein Body nötig

- Weiterleitung auf Collection (optional) -> HATEAOS (später)

OPTIONS

- idempotent wie GET, ohne Side-Effect
- liefert Informationen über mögliche Methode
- kann von Authorisation Status abhängig sein
- Wenn ausgelogged: GET
- Wenn eingelogged: POST, PUT
- Wenn spezielle Rolle: DELETE

HEAD

- benahe identisch zu GET
- Server braucht aber kein Body zurück liefern

4 Constraints

- es geht im Grunde um 4 Constraints - haben sich heraus kristalisiert

1.
Adressierbarkeit

- jede resource muss eindeutig im Netz erreichbar sein
- hinter jeder URL muss sich immer die selbe Resource verbergen

2.
Repräsentationen

- Abwandlung einer Resource kann nur eine Repräsentation sein
Repräsentaton kann sein:
- andere Sprache
- anderer Content-Type

3.
Zustandslosigkeit

- jeder Service bekommt immer alle informationen um einen Request zu bearbeiten
- Service weiß nichts von vorhergehenden Requests
- Widerspricht eigentlich unserer Denkweise von Sessions/Cookies
- erhört dann natürlich Skalierbarkeit, da Lastverteilung einfacher
- Load Balancer muss kann nach belieben Requests auf verschiedene Nodes verteilen
-> Client App. ist muss Zustand halten

4.
Operationen

- dienen dem state transfer
- Rudimentärer Bestandteil von REST
- Resource kann nur über eine Operation, veändert, angelegt oder gelöscht werden
- nicht in der URL!!
